# API Reference

*Public endpoints for querying, authentication, and administration*

This document describes the HTTP API exposed by NAM for external consumers. Internal service-to-service APIs are not covered here.

---

## Overview

NAM exposes a single external API service (the **Query Service**) that handles:

* Authentication and user management
* Semantic queries
* System administration
* Document browsing
* Data replication management
* Storage management

All endpoints use JSON request/response bodies and follow REST conventions.

---

## Authentication

NAM uses **JWT (JSON Web Token)** authentication. When authentication is enabled:

* All API endpoints (except login and health checks) require a valid JWT Bearer token
* Tokens are obtained via the login endpoint
* Tokens carry identity and role claims (user or admin)
* Tokens are time-limited and can be refreshed before expiry
* Admin endpoints require `admin` role in the token claims

### Login

```
POST /v1/auth/login
```

Authenticates a user and returns a signed JWT token.

**Request body:**
```json
{
  "username": "string",
  "password": "string"
}
```

**Response:**
```json
{
  "token": "eyJ...",
  "expires_in": 3600
}
```

The returned token should be included in subsequent requests as a Bearer token:
```
Authorization: Bearer eyJ...
```

### Refresh Token

```
POST /v1/auth/refresh
```

Returns a new token with an extended expiry. Requires a valid (non-expired) token.

**Response:**
```json
{
  "token": "eyJ...",
  "expires_in": 3600
}
```

### Change Password

```
PUT /v1/auth/password
```

Changes the authenticated user's password. Requires a valid token.

**Request body:**
```json
{
  "old_password": "string",
  "new_password": "string"
}
```

**Response:**
```json
{
  "ok": true,
  "message": "Password updated"
}
```

### Reset Password

```
POST /v1/auth/reset-password
```

Resets a user's password using a single-use reset token (generated by an admin).

**Request body:**
```json
{
  "token": "string",
  "new_password": "string"
}
```

**Response:**
```json
{
  "ok": true
}
```

## Query

### Execute Query

```
POST /v1/query
```

Executes a semantic query against NAM's address space. Requires a valid token.

**Request body:**
```json
{
  "query": "string",
  "limit": 10,
  "namespace": null,
  "hints": null
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `query` | string | Yes | Natural language query text |
| `limit` | integer | No | Maximum results to return (default: 10) |
| `namespace` | string | No | Restrict query to a specific namespace |
| `hints` | object | No | Additional hints for the query planner |

**Response:**
```json
{
  "ok": true,
  "mode": "EXPLORATORY",
  "affordance": null,
  "results": [
    {
      "record_id": "string",
      "data_pointer": "string",
      "payload": { ... },
      "addresses_matched": [ ... ]
    }
  ],
  "truncated": false,
  "errors": []
}
```

| Field | Type | Description |
|-------|------|-------------|
| `ok` | boolean | Whether the query executed successfully |
| `mode` | string | Query mode used: `EXPLORATORY` or `AFFORDANCE` |
| `affordance` | string or null | Detected affordance verb, if any |
| `results` | array | Matching records with payloads |
| `truncated` | boolean | Whether results were limited by the `limit` parameter |
| `errors` | array | Any non-fatal errors encountered during execution |

**Query modes:**

* **EXPLORATORY** — No affordance verb detected. Probes broadly across ontology tiers.
* **AFFORDANCE** — Affordance verb recognized (e.g., "Where is...", "Who leads..."). Probes narrowly using affordance-specific strategies.

### Graph Walk

```
POST /v1/graph/walk
```

Traverses NAM's graph index starting from a query, following semantic relationships across dimensions. Requires a valid token.

**Request body:**
```json
{
  "query": "string",
  "hop_depth": 2,
  "limit": 20
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `query` | string | Yes | Starting query text |
| `hop_depth` | integer | No | Number of hops to traverse (default: 2) |
| `limit` | integer | No | Maximum results per hop (default: 20) |

**Response:**
```json
{
  "ok": true,
  "hops": [
    {
      "depth": 0,
      "results": [ ... ]
    },
    {
      "depth": 1,
      "results": [ ... ]
    }
  ]
}
```

### Record Walk

```
POST /v1/graph/record-walk
```

Traverses the graph index starting from a specific record, discovering semantically related records through shared address dimensions. Requires a valid token.

**Request body:**
```json
{
  "record_id": "string",
  "hop_depth": 1,
  "limit": 20
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `record_id` | string | Yes | Source record identifier |
| `hop_depth` | integer | No | Number of hops (default: 1) |
| `limit` | integer | No | Maximum results per hop (default: 20) |

### Entity Documents

```
GET /v1/graph/entity/{entity_id}/documents?limit=20
```

Returns documents associated with a specific entity, resolved through the graph index.

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `entity_id` | path | Yes | Entity identifier (hash-based) |
| `limit` | query | No | Maximum documents to return (default: 20) |

### Graph Statistics

```
GET /v1/graph/stats
```

Returns statistics about the graph index: total entries, index types, and edge counts.

### Document Browsing

#### List Documents

```
GET /v1/admin/docs/{bucket}?limit=100&offset=0
```

Lists document keys in a bucket. Uses CDC-based key scanning.

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `bucket` | path | Required | Bucket name to browse |
| `limit` | query | 100 | Maximum keys to return |
| `offset` | query | 0 | Number of keys to skip |

**Response:**
```json
{
  "bucket": "main",
  "keys": ["wiki::Alabama", "wiki::Anarchism", ...],
  "count": 100,
  "total": 1000
}
```

#### Get Document

```
GET /v1/admin/docs/{bucket}/{key}
```

Retrieves a single document by key.

**Response:**
```json
{
  "bucket": "main",
  "key": "wiki::Alabama",
  "value": { ... },
  "cas": 12345678
}
```

## KV Proxy

NAM optionally exposes a **KV proxy** that provides authenticated direct access to the data service's key-value protocol.

The KV proxy is a TCP service (not HTTP). Connection flow:

1. Client connects to the KV proxy port
2. Client sends a JWT token as the first line (newline-terminated)
3. Proxy validates the token
4. If valid, proxy responds with `OK\n` and begins bidirectional forwarding to the upstream data service
5. If invalid, proxy responds with `ERR invalid token\n` and closes the connection

### VBucket Map

```
GET /v1/kv/map?bucket=main
```

Returns the vbucket map for the KV proxy, with addresses translated for external access. This is used by SDK clients to determine which partition a key maps to.

Requires a valid token.

## Error Handling

All endpoints return structured error responses:

```json
{
  "ok": false,
  "error": "Error description",
  "detail": "Additional context"
}
```

HTTP status codes follow standard conventions:

| Code | Meaning |
|------|---------|
| 200 | Success |
| 201 | Created |
| 202 | Accepted (async operation started) |
| 400 | Bad request (invalid parameters) |
| 401 | Unauthorized (missing or invalid token) |
| 403 | Forbidden (insufficient role) |
| 404 | Not found |
| 409 | Conflict (operation already in progress) |
| 429 | Rate limited |
| 500 | Internal server error |
| 503 | Service unavailable |

---

## Rate Limiting

When enabled, rate limiting applies to:

* Authentication attempts (login endpoint)
* Query volume (per-user)
* Administrative operations

Rate limit headers are included in responses when limits are approached.

---

## Summary

NAM's public API provides:

* **Authentication** — JWT-based, role-scoped, time-limited
* **Querying** — Single endpoint for all semantic queries, with automatic mode detection
* **Graph traversal** — Graph walk, record walk, and entity document retrieval via the graph index
* **Administration** — User management, system status, pipeline control, document browsing
* **Monitoring** — Bucket statistics, storage statistics, time-series metrics, pipeline error breakdown
* **Data replication** — Cross-datacenter replication management
* **Storage management** — S3 backup configuration and savepoints
* **Direct data access** — Authenticated KV proxy for SDK integration

All endpoints are deterministic — the same request with the same data produces the same response.
